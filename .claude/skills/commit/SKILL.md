Stage and commit changes following conventional commit format.

## Arguments

Optional arguments the user may provide:
- **type**: Commit type (e.g., `feat`, `fix`, `chore`). If not provided, infer from the changes.
- **scope**: Module scope (e.g., `workspace`, `auth`). If not provided, infer from changed files.
- **message**: Description text. If not provided, generate from the diff.

## Steps

### 1. Check state

```bash
git diff --cached --stat && echo "---" && git diff --stat && echo "---" && git ls-files --others --exclude-standard
```

If nothing to commit, tell the user and stop.

### 2. Size guard

Before staging, count how many files and lines are involved:

```bash
git diff --stat HEAD 2>/dev/null | tail -1
```

**Hard limits:**
- More than **15 files changed** → STOP. Tell the user: "This commit touches N files — that's too broad for a single commit. Split it into focused commits by feature/concern first."
- More than **400 lines changed** (insertions + deletions combined) → STOP with the same message.

If either limit is exceeded, suggest a logical split based on the changed file paths and wait for the user to act. Do NOT proceed with a large commit.

### 3. Stage changes

If nothing is staged, run `git add -A` to stage everything. If something is already staged, commit what's staged as-is.

Re-run the size guard on the staged diff after staging:

```bash
git diff --cached --stat | tail -1
```

Apply the same hard limits to the staged set.

### 3. Build the commit message

Get the diff:
```bash
git diff --cached
```

**Type** — infer from changes if not provided:
- New features → `feat`
- Bug fixes → `fix`
- Tests only → `test`
- Config/deps/build → `chore`
- Restructure without behavior change → `refactor`
- Docs only → `docs`
- Style/formatting → `style`
- Performance → `perf`

**Scope** — infer from changed files (e.g., `workspace`, `auth`, `api`, `hooks`, `components`). Omit if changes span many unrelated areas. Lowercase + hyphens only.

**Description** — concise, lowercase, no period.

Format: `type(scope): description`

**Subject line hard limit: 72 characters total** (including `type(scope): `).
- Count the characters before writing the commit.
- If the natural description exceeds the limit, shorten it — drop adjectives, abbreviate scope, compress wording. Never truncate mid-word with `...`.
- If a body is needed for extra context, add it after a blank line (no length limit on body lines, but wrap at 72 chars per line).

### 4. Commit

```bash
git commit -m "{commit_message}"
```

If the commit fails, show the error and help fix it.

### 5. Output

Print:
```
✔ {short_hash} {commit_message}
```

Then ask if they want to push.

**NEVER add AI co-author, "generated by", "assisted by", or any AI attribution to commit messages.**
